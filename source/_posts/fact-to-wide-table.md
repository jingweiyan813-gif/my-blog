> **本文要点**
>
> 数据进入 ODS 后，并不等于已经成为可分析的数据资产。数仓建设需要先通过分层明确数据生产职责，再通过建模表达业务过程、事实粒度、维度上下文和指标口径。本文从数据开发工程视角，系统说明数据分层怎么设计、数据建模怎么做，以及每一类模型的工程边界。

本文聚焦两个问题：

```text
1. 数据进入 ODS 后，如何进行数仓分层？
2. 如何通过数据建模把原始记录变成分析资产？
```

不展开实时数仓、湖仓架构、调度平台和具体存储引擎实现。

全文按数据生产链路展开：先讲分层边界，再讲建模方法，再讲事实表、维度表、汇总表、宽表和指标如何落到每一层，最后用一个支付业务过程把这些设计选择串起来。

## 1. 数据分层：按职责组织数据生产链路

不同公司对层级命名会有差异。经典口径常用 ODS-DWD-DWS-ADS 四层；更细的体系会拆出 DIM、DWM、DM、APP 等层级。

命名可以不同，但职责划分通常稳定。这也是行业里讨论分层时真正需要对齐的部分：不是名称完全一致，而是每一层承担的职责边界一致。

如果没有分层，接入、清洗、建模、指标和报表逻辑会混在同一批 SQL 里。短期看出数很快，长期会带来口径变更困难、问题排查困难和下游重复建设。

可以先用一张表把分层链路看清楚：

| 层级 | 输入 | 主要处理 | 输出 | 主要使用方 |
|---|---|---|---|---|
| ODS | 业务库、日志、消息、文件、API | 接入、落分区、保留源端语义 | 贴源表、原始日志表、变更流水表 | DWD、DIM、排障回溯 |
| DWD | ODS 明细数据 | 清洗、标准化、按业务过程沉淀明细数据 | 明细事实表、标准化明细表 | DWS/DWM、DM、明细分析 |
| DIM | ODS/DWD 中的实体数据 | 统一维度属性、处理层级和历史变化 | 一致性维度表、快照维表、拉链表 | DWD、DWS/DWM、DM、ADS |
| DWS/DWM | DWD + DIM | 轻度汇总、原子指标口径、公共指标计算结果 | 公共汇总表、主题汇总表 | DM、ADS、报表、接口 |
| DM | DWS/DWM + DWD + DIM | 面向主题或场景组织宽表 | 主题宽表、分析数据集 | 分析师、BI、数据产品 |
| ADS/APP | DM 或公共汇总表 | 应用裁剪、结果固化、接口适配 | 报表表、接口表、应用结果表 | 业务报表、服务接口、应用系统 |

从工程角度看，分层的核心是控制三件事：

```text
数据语义从原始到业务逐步清晰；
计算逻辑从公共到个性逐步收敛；
下游问题能沿血缘逐层回溯。
```

### 1.1 ODS：贴源和可追溯

ODS 是数仓入口层，目标是把源端数据稳定接入分析侧，并保留可回溯的原始语义。

ODS 的输入通常包括：

```text
业务数据库表
埋点日志
服务日志
消息队列
对象存储文件
第三方 API 数据
CDC 变更记录
```

ODS 的输出通常是贴源表或原始日志表，例如：

```text
ods_order_order_df
ods_trade_payment_di
ods_app_event_log_di
ods_user_profile_df
```

ODS 表名通常能看出来源系统、源表或数据类型。它不追求业务抽象，而追求源端可追溯。

设计要点：

- 保留源端主键、状态、更新时间、删除标识；
- 增加同步时间、批次号、来源、操作类型等技术字段；
- 明确分区口径，是同步日期、事件日期还是业务日期；
- 异常数据尽量保留并标记，不在入口层随意过滤；
- 支持重跑、补数、回刷和源端对账。

ODS 可以做少量技术处理：

| 可以做 | 不建议做 |
|---|---|
| 字符集处理、基础格式处理、分区生成、同步字段补充 | 复杂清洗、业务状态标准化、指标计算、跨源 join |

ODS 的边界是：只做必要技术处理，不做复杂业务口径。

如果在 ODS 层直接做指标计算，接入逻辑和业务逻辑会耦合在一起。后续源端字段变化、状态枚举变化、迟到数据补写，都会影响指标口径解释。

### 1.2 DWD：明细数据和业务规则下沉

DWD 是 ODS 之后的核心业务语义层。它不是单纯清洗层，而是按业务过程沉淀稳定、可复用的明细数据层，其中最常见的产物是明细事实表。

DWD 的输入主要来自 ODS，也可能引用 DIM 做必要维度补充。

DWD 的输出通常是稳定的明细事实表或标准化明细表，例如：

```text
dwd_trade_order_detail_di
dwd_trade_payment_detail_di
dwd_user_login_detail_di
dwd_app_event_detail_di
```

这些表不再只是源表复制，而是围绕明确业务过程设计。

DWD 需要做：

| 工作 | 说明 |
|---|---|
| 清洗标准化 | 时间、金额、枚举、布尔值、空值统一 |
| 粒度统一 | 一张明细表只表达一种主粒度 |
| 业务规则下沉 | 通用状态判断、有效性判断、异常标记 |
| 退化维度 | 保留订单号、流水号等不单独建维表的业务标识，支持明细追溯 |
| 源端变化屏蔽 | 源表变更后尽量保持 DWD 语义稳定 |

DWD 的设计关键是粒度。

```text
dwd_trade_payment_detail_di
一行 = 一次支付流水

dwd_user_login_detail_di
一行 = 一次登录事件

dwd_app_event_detail_di
一行 = 一次用户行为事件
```

DWD 的边界是：沉淀通用明细和共性业务规则，不沉淀某个报表专用口径。如果某个规则只服务一个看板或一个活动分析，通常不应进入公共 DWD。

### 1.3 DIM：一致维度

DIM 负责提供分析上下文。事实表回答“发生了什么”，维度表回答“在什么背景下发生”。

DIM 的输入可以来自 ODS，也可以来自 DWD 或人工维护的维度配置。

DIM 的输出是维度表，例如：

```text
dim_user_info_df
dim_product_info_df
dim_payment_channel_df
dim_region_df
dim_date_df
```

常见维度包括：

```text
用户维度
商品维度
渠道维度
地区维度
组织维度
时间维度
实验分组维度
实验维度
```

DIM 的设计原则：

- 维度主键稳定；
- ID 和 name 成对出现；
- 层级维度支持上钻和下钻；
- 同一业务含义字段在不同模型中保持同名同义；
- 核心维度和扩展维度分离；
- 聚合指标通常不要放入维表。

DIM 还要处理历史变化。例如用户等级、渠道名称、组织归属、商品类目会变化。

| 处理方式 | 说明 | 适用场景 |
|---|---|---|
| Type 0 | 属性写入后不再变化 | 首次注册渠道、出生日期 |
| Type 1 | 新值覆盖旧值 | 只关心当前状态 |
| Type 2 / 拉链表 | 新增一行记录变化，维护生效时间 | 需要按历史时点还原 |
| 快照维表 | 每天保留一份全量快照 | 离线分析中常用，使用简单 |

DIM 的边界是：维护实体属性和上下文口径，不承担事实聚合。

### 1.4 DWS/DWM：公共汇总和指标结果沉淀

DWS 或 DWM 主要负责公共汇总。不同团队对这两层命名不同：如果同时保留，DWM 通常更偏中间公共汇总，DWS 更偏面向主题域的服务汇总；如果体系较简化，也可以只保留 DWS。本文把它们合并讨论，重点看它们在明细数据和应用出口之间承担的公共汇总职责。

DWS/DWM 的输入主要来自 DWD 和 DIM。

DWS/DWM 的输出通常是按公共粒度汇总的事实表，例如：

```text
dws_trade_user_payment_1d_di
dws_trade_channel_payment_1d_di
dws_user_active_1d_di
dws_app_event_stats_1d_di
```

它们的共同职责是：

- 沉淀原子指标的计算口径和公共汇总结果；
- 沉淀公共衍生指标的计算结果；
- 做轻度汇总；
- 减少下游重复计算；
- 为 DM、ADS、报表和接口提供稳定输入。

例如支付分析中，可以在 DWS/DWM 沉淀：

```text
pay_request_cnt
pay_success_cnt
pay_fail_cnt
pay_amount
```

而不是让每个报表从 DWD 明细层重复聚合。

DWS/DWM 的设计关键是公共粒度。

```text
用户 + 日期
渠道 + 日期
商品 + 日期
页面 + 日期
```

如果某个汇总只服务一个临时报表，它更适合放在 ADS/APP，而不是进入公共汇总层。

### 1.5 DM 与 ADS/APP：分析出口和应用出口

DM 面向分析主题和数据集市，通常通过事实、维度和公共汇总形成宽表或数据集。

ADS/APP 面向具体应用，通常服务报表、接口、数据产品、可视化大屏或应用系统。

DM 和 ADS/APP 经常被混在一起，但它们的职责不同。

| 层级 | 适合做什么 | 风险 |
|---|---|---|
| DM | 主题宽表、场景数据集、多维分析表 | 宽表膨胀、字段边界失控 |
| ADS/APP | 应用裁剪、报表结果、接口输出 | 公共逻辑沉淀过多导致口径分散 |

DM 的典型输出：

```text
dm_user_trade_profile_df
dm_channel_payment_analysis_df
dm_product_sales_summary_df
```

ADS/APP 的典型输出：

```text
ads_payment_dashboard_1d
app_user_profile_api_df
ads_user_active_dashboard_1d
```

宽表不是建模起点，而是建模后的分析出口。宽表可以提高查询效率，但不能替代 DWD 明细数据和 DIM 一致维度。

## 2. 数据建模：先理解业务，再选择模型

实际业务里的数据建模，不是只套一种模型。Kimball 维度建模在分析型数仓中非常常用，但它不是唯一方法。业务稳定性、源系统复杂度、历史追溯要求、指标治理要求不同，建模方式也会不同。

更贴近实际的建模流程是：

```text
1. 做业务调研，明确业务流程和分析问题
2. 划分业务域、主题域和核心对象
3. 识别业务过程和事件
4. 声明数据粒度
5. 选择建模方法
6. 设计事实、维度、关系和指标
7. 落到 DWD、DIM、DWS/DWM、DM、ADS 各层
```

### 2.1 先做业务调研

建模前需要先理解业务，而不是先看源表。

业务调研至少要问清楚：

| 调研项 | 需要确认的问题 |
|---|---|
| 业务流程 | 业务从开始到结束有哪些关键节点 |
| 核心对象 | 用户、订单、商品、任务、渠道、设备等对象是什么 |
| 业务事件 | 哪些动作会产生数据，例如下单、支付、退款、点击、反馈 |
| 分析问题 | 下游要看趋势、转化、归因、质量、成本还是异常 |
| 指标口径 | 指标分子、分母、时间口径、排除规则是什么 |
| 历史要求 | 是否需要还原历史状态，是否需要回溯变更过程 |
| 使用方式 | 是 BI 查询、接口服务、算法样本，还是数据产品 |

如果跳过业务调研，模型往往会变成“源表搬家”：表名变了，字段标准化了，但业务过程、粒度和指标口径仍然不清楚。

### 2.2 划分业务域、主题域和核心对象

业务域是较粗的业务边界。主题域是在业务域内进一步抽象出的分析主题。

示例：

```text
交易域
  - 订单主题
  - 支付主题
  - 退款主题
  - 履约主题

用户域
  - 注册主题
  - 活跃主题
  - 留存主题
  - 画像主题

流量域
  - 访问主题
  - 点击主题
  - 转化主题
  - 实验主题
```

这一步解决的是数据资产目录问题。没有业务域和主题域，模型会按临时需求散落；后续找数、复用、治理都会困难。

### 2.3 识别业务过程

业务过程是可被观察、可被度量的业务动作。在维度建模中，事实表通常围绕业务过程设计。

判断一个业务过程是否值得建模，可以看三点：

| 判断标准 | 说明 |
|---|---|
| 是否是明确业务动作 | 下单、支付、退款、登录、点击、反馈 |
| 是否产生可度量结果 | 金额、次数、耗时、状态、成本 |
| 是否会被独立分析 | 是否需要统计、对比、追溯、归因 |

例如交易域中，订单主表、支付流水表、退款表是源表；但建模时关注的是下单、支付、退款这些业务过程。

流量分析中，源数据可能是埋点日志、访问日志、实验配置和转化结果表；但建模时关注的是访问、点击、曝光、转化这些业务过程。

### 2.4 声明粒度

粒度定义事实表中一行数据代表什么。

这是事实表设计中最重要的约束。

| 粒度 | 一行代表什么 | 适用场景 |
|---|---|---|
| 订单粒度 | 一个订单 | 订单整体分析 |
| 子订单粒度 | 一个订单明细 | 商品、类目、商家分析 |
| 支付流水粒度 | 一次支付请求 | 支付成功率、支付渠道分析 |
| 用户日粒度 | 一个用户一天 | 活跃、留存、消费分层 |
| 行为事件粒度 | 一次曝光、点击或访问 | 流量分析、转化分析 |
| 任务流转粒度 | 一个任务从开始到结束的过程 | 履约、工单、审核分析 |

不同粒度不要强行混在一张事实表中。粒度混用会造成重复计数、金额放大、join 膨胀和指标口径不可解释。

### 2.5 选择建模方法

实际项目里通常会组合多种建模思想。

| 建模方法 | 适合场景 | 常见产物或用途 |
|---|---|---|
| ER / 三范式建模 | 核心实体关系稳定，需要减少冗余 | 主数据、核心实体关系、底层规范化模型 |
| Kimball 维度建模 | 面向分析，强调事实表、维度表和查询效率 | 事实表、维度表、一致性维度、总线矩阵 |
| Data Vault 建模 | 多源异构、关系复杂、强历史追溯 | Hub、Link、Satellite，用于业务键、关系和属性变化追踪 |
| 宽表建模 | 查询高频、分析路径稳定 | DM、ADS、数据集市 |
| OneData 指标建模 | 指标口径治理、统一指标出口 | DWS/DWM、指标平台、ADS |

这几种方法不是互斥关系。一个实际数仓可能在底层用规范化思想组织核心实体，在 DWD/DIM 使用维度建模，在 DWS/DWM 沉淀公共指标计算结果，在 DM/ADS 形成宽表出口。

后文以维度建模为主线，其他方法只作为复杂场景下的补充判断。本文不展开 Data Vault 的完整模型结构，只强调它适合多源整合、业务键统一和强历史追溯场景。

选择建模方法时，可以按业务复杂度判断：

| 业务情况 | 推荐做法 | 忽视后的代价 |
|---|---|---|
| 单一源系统、分析需求明确 | 以 Kimball 维度建模为主，配合指标口径管理 | 过度设计，建设成本高 |
| 多源系统、同一对象在多系统存在 | 先统一业务键和主数据，再做事实和维度 | 同一个用户、商品、订单无法对齐 |
| 状态流转复杂，要求历史可追溯 | 保留明细事件，必要时引入规范化或 Data Vault 思路 | 容易只看到当前结果，难以解释过程变化 |
| 指标争议多、复用要求高 | 建设原子指标、修饰词、派生指标体系 | 每个报表各算一套，指标长期不一致 |
| 查询路径固定、性能压力大 | 在 DM/ADS 做宽表或结果表 | 用户查询成本高，应用响应慢 |

所以，“只说一种模型”通常不够。但也不是每个项目都要把所有方法上齐。对于大多数分析型离线数仓，主线仍然可以是维度建模：围绕业务过程建设事实表，围绕分析上下文建设维度表，再通过 DWS/DWM 和指标体系沉淀公共口径与公共计算结果。其他建模方法更多用于补足边界，例如多源整合、强历史追溯、主数据治理和高频应用出口。

### 2.6 设计事实、维度和指标

如果采用维度建模，核心设计动作包括确定事实、维度和指标。

事实是业务过程产生的度量。

| 类型 | 示例 | 设计建议 |
|---|---|---|
| 可加事实 | 支付金额、访问次数、点击次数 | 可以在明细层保存 |
| 半可加事实 | 库存、余额、累计状态 | 需要结合时间周期解释 |
| 不可加事实 | 成功率、转化率、汇率 | 保存分子分母，不直接保存比率 |

维度描述事实发生的上下文。

```text
时间
用户
商品
渠道
地区
组织
实验分组
```

指标是事实在一定时间周期、修饰条件和统计粒度下形成的业务口径。

```text
派生指标 = 原子指标 + 修饰词 + 时间周期
```

例如支付请求成功率，应拆成支付成功请求数和支付请求数，而不是在明细层直接保存一个成功率字段。

## 3. 建模结果如何落到具体模型表

前面讲的是建模方法，落到数仓开发中，最终要体现为事实表、维度表、汇总表和宽表。它们分别承接不同层级职责，不能互相替代。

模型类型和分层之间不是一一对应关系，但通常会形成这样的落点：

| 模型类型 | 常见层级 | 主要作用 |
|---|---|---|
| 明细事实表 | DWD | 承接业务过程和明细粒度 |
| 一致维度表 | DIM | 提供可复用的分析上下文 |
| 公共汇总表 | DWS/DWM | 沉淀公共粒度和指标计算结果 |
| 主题宽表 | DM | 降低分析和取数成本 |
| 应用结果表 | ADS/APP | 服务报表、接口和具体数据产品 |

### 3.1 事实表类型

常见事实表有三类。

| 类型 | 适合场景 | 特点 |
|---|---|---|
| 事务事实表 | 离散事件，如下单、支付、点击、退款 | 追加为主，粒度最细 |
| 周期快照事实表 | 周期状态，如库存、账户余额、用户日状态 | 按固定周期记录状态 |
| 累计快照事实表 | 长流程，如订单履约、工单流转、任务执行 | 一行记录多个节点时间，可能更新 |

选择方法：

```text
如果关注事件发生了多少次，用事务事实表。
如果关注某个时间点的状态，用周期快照事实表。
如果关注对象从开始到结束的流转过程，用累计快照事实表。
```

### 3.2 维度表和缓慢变化维

维度并不总是静态的。用户等级、渠道名称、组织归属、商品类目都可能变化。

常见处理方式：

| 处理方式 | 说明 | 适用场景 |
|---|---|---|
| Type 0 | 属性写入后不再变化 | 首次注册渠道、出生日期 |
| Type 1 | 新值覆盖旧值 | 只关心当前状态 |
| Type 2 / 拉链表 | 新增一行记录变化，维护生效时间 | 需要按历史时点还原 |
| Type 3 | 增加历史字段保存有限历史值 | 只关心当前值和上一个值 |
| 快照维表 | 每天保留一份全量快照 | 离线分析中常用，使用简单 |

选择哪一种，取决于维度变化频率、查询成本、存储成本，以及是否需要严格还原历史时点。

### 3.3 总线矩阵

总线矩阵用于连接业务过程和一致性维度。

示例：

| 业务过程 / 维度 | 用户 | 商品 | 渠道 | 地区 | 时间 | 实验分组 |
|---|---|---|---|---|---|---|
| 下单 | Y | Y | Y | Y | Y |  |
| 支付 | Y | Y | Y | Y | Y |  |
| 退款 | Y | Y | Y |  | Y |  |
| 访问 | Y |  | Y |  | Y | Y |
| 点击 | Y | Y | Y |  | Y | Y |
| 转化 | Y | Y | Y | Y | Y | Y |

总线矩阵的作用：

- 明确哪些业务过程共享维度；
- 提前发现维度口径冲突；
- 约束事实表之间的可关联性；
- 为后续宽表和指标体系提供基础。

### 3.4 汇总表和宽表

DWS/DWM 汇总表主要沉淀公共指标计算结果和轻度汇总。

设计汇总表时需要明确：

```text
汇总粒度
维度组合
原子指标
时间周期
是否可复用
```

DM 宽表面向分析主题和消费场景。

宽表设计需要明确：

```text
主粒度
主事实表或主汇总表
关联哪些一致维度
哪些字段进入核心宽表
哪些字段进入扩展表
指标是否可追溯
```

宽表的价值是降低分析复杂度，不是无限堆字段。

## 4. 指标体系：建模结果的业务出口

指标体系不是脱离模型单独存在的一套口径文档，而是建模结果的业务出口。事实表提供分子、分母和度量，维度表提供分析上下文，DWS/DWM 沉淀公共汇总结果，DM/ADS 面向具体消费场景输出指标结果。

因此，指标管理要回到三个建模问题：

```text
指标基于哪个业务过程？
指标使用什么事实粒度？
指标可以按哪些一致维度分析？
```

一个指标至少需要定义：

| 口径项 | 示例 |
|---|---|
| 指标名称 | 支付请求成功率 |
| 业务过程 | 支付请求 |
| 统计粒度 | 支付流水 |
| 原子指标 | 支付成功请求数、支付请求数 |
| 修饰词 | 新用户、某渠道、某实验分组 |
| 时间周期 | 最近 1 天、最近 7 天、自然月 |
| 计算方式 | 支付成功请求数 / 支付请求数 |
| 数据来源 | DWD 明细表、DWS 汇总表、DM 宽表 |
| 可选维度 | 渠道、地区、用户类型、日期 |

这里体现 OneData 思想：

```text
派生指标 = 原子指标 + 修饰词 + 时间周期
```

工程实现上，要优先保存可加的基础度量，而不是直接固化不可加比率。例如支付请求成功率可以展示在 ADS/DM，但在公共汇总层更适合保存支付请求数和支付成功请求数。否则按天、渠道或地区再次汇总时，直接平均成功率会得到错误结果。

例如支付请求成功率不应只定义为：

```text
支付成功率 = 成功数 / 总数
```

而应明确：

```text
业务过程：支付请求
粒度：支付流水
分子：支付成功请求数
分母：支付请求数
时间口径：支付请求时间
排除规则：测试订单、压测流量、无效支付单是否剔除
```

指标还需要血缘：

```text
ADS/DM 指标
  -> DWS/DWM 原子指标口径和公共汇总结果
  -> DWD 明细数据
  -> ODS 源数据
```

没有血缘的指标，一旦异常就很难排查。

## 5. 示例：支付业务过程如何完成分层与建模

为了把上面的分层、建模和指标口径串起来，可以看一个支付业务过程的建模示例。

支付分析通常会关心三类问题：

```text
支付请求有没有发起？
支付请求是否成功？
不同渠道、地区、用户类型下的支付表现是否稳定？
```

这些问题决定了模型不能直接从订单表上计算成功率，也不能为单个看板临时拼一张结果表。更稳妥的做法是先识别支付请求这个业务过程，再按分层把明细、维度、汇总和应用出口拆开。

建模链路：

| 层级 | 建模动作 | 关键选择 | 设计原因 |
|---|---|---|---|
| ODS | 接入订单、支付流水、回调日志、用户、渠道等源数据 | 保留源端表结构和原始状态 | 支持对账、补数、排障和回溯 |
| DWD | 建设支付流水明细事实表 | 一行表示一次支付请求 | 成功率应基于支付请求粒度，而不是订单粒度 |
| DIM | 建设用户、渠道、地区、时间等一致维度 | 统一维度主键和属性口径 | 支持按维度分析，避免下游各自 join 和解释 |
| DWS/DWM | 汇总支付请求数、支付成功数、支付失败数 | 保存分子分母，不保存直接比率 | 成功率不可加，直接汇总会导致口径错误 |
| DM/ADS | 输出渠道支付分析宽表或支付看板结果表 | 面向查询和报表裁剪字段 | 服务消费场景，但保留回溯到 DWD 的能力 |

DWD 支付明细事实表可以声明为：

```text
表名：dwd_trade_payment_detail_di
业务过程：支付请求
粒度：一次支付流水
主键：payment_id
分区：p_date，按支付请求时间落分区
```

核心字段：

| 字段 | 说明 |
|---|---|
| payment_id | 支付流水 ID |
| order_id | 订单 ID |
| user_id | 用户 ID |
| channel_id | 支付渠道 ID |
| pay_request_time | 支付请求时间 |
| pay_amount | 支付金额，单位分 |
| pay_status | 标准化支付状态 |
| is_success | 是否支付成功，0/1 |

这里没有把订单金额、订单状态等订单粒度字段直接作为主事实，是为了避免支付重试时产生金额重复。订单信息可以通过 order_id 回溯，也可以在后续宽表中按消费场景选择性冗余。

这个例子的关键不是成功率公式，而是不要把订单粒度、支付流水粒度和报表口径混在一张表里。如果一个订单存在多次支付重试，订单金额、支付次数和成功率就可能被重复计算。实际建模时，应把订单、支付、退款拆成不同业务过程，再通过订单 ID、用户 ID、渠道 ID、时间等一致维度建立可分析关系。

这个例子对应了参考资料中的几个方法：

| 参考方法 | 在例子中的体现 |
|---|---|
| Kimball 四步骤 | 选择支付业务过程、声明支付流水粒度、确定渠道/用户等维度、确定支付金额/次数等事实 |
| 总线矩阵 | 支付、退款、订单等业务过程共享用户、渠道、时间等一致维度 |
| SCD / 快照维表 | 用户等级、渠道名称、地区归属等维度变化处理 |
| 公共逻辑下沉 | 支付状态标准化、金额单位统一、异常标记沉淀到 DWD |
| OneData 指标思想 | 用原子指标、修饰词、时间周期定义支付请求成功率 |

## 6. 落地建议与检查表

数仓分层和建模不建议一次性铺满所有主题。

更稳妥的实施路线：

| 阶段 | 目标 | 主要工作 |
|---|---|---|
| 基础接入 | 建立稳定入口 | 确定数据源接入方式，建设 ODS，配置基础质量监控，制定命名和字段规范 |
| 核心建模 | 沉淀公共模型 | 选择 1-2 个核心业务过程，建设 DWD 明细表、DIM 维表、DWS/DWM 公共汇总 |
| 应用扩展 | 形成数据出口 | 建设 DM/ADS 宽表和应用表，对接 BI、接口、数据产品，补齐血缘和影响分析 |

常见问题：

| 问题 | 典型表现 | 处理方式 |
|---|---|---|
| 数据孤岛 | 不同主题表无法关联 | 建立统一业务域、主题域和一致性维度 |
| 口径不一致 | 同一指标在不同报表结果不同 | 定义原子指标、派生指标和指标来源 |
| 宽表失控 | 字段持续膨胀，查询变慢 | 核心表和扩展表拆分，低频字段拆出 |
| ADS 层过重 | 报表 SQL 同时承担清洗、join、指标计算 | 公共逻辑下沉到 DWD、DWS/DWM 或 DM |
| 维护困难 | 源表变更后影响范围不清 | 完善血缘、表说明、字段说明和变更流程 |
| 性能瓶颈 | 查询响应慢、资源消耗高 | 预聚合、合理分区、控制 join、建设主题宽表 |

这些问题本质上都不是单纯 SQL 问题，而是分层边界、模型粒度、公共口径和治理能力没有同步设计。

上线核心模型表前，可以按以下检查。

| 检查项 | 需要确认的问题 |
|---|---|
| 层级定位 | 表应该在 ODS、DWD、DIM、DWS/DWM、DM 还是 ADS |
| 业务过程 | 是否围绕明确业务事件设计 |
| 粒度 | 一行数据代表什么，是否只有一种粒度 |
| 主键 | 主键或唯一键是否能支撑去重、回刷和 join |
| 事实 | 度量是否可解释、可汇总，是否避免过早保存比率 |
| 维度 | 是否使用一致性维度，维度属性是否稳定 |
| 汇总口径 | 汇总粒度、维度组合、时间周期是否明确 |
| 指标 | 原子指标、修饰词、时间周期、统计方法是否明确 |
| 公共逻辑 | 通用清洗和指标逻辑是否下沉 |
| 血缘 | 下游是否能追溯到来源表和加工逻辑 |

## 7. 小结

数据进入 ODS 后，真正的数据开发工作才开始。

数仓分层要回答：

```text
每一层承担什么职责？
哪些逻辑应该下沉？
哪些逻辑只属于应用出口？
层间依赖如何保持清晰？
```

数据建模要回答：

```text
业务过程是什么？
事实表粒度是什么？
事实和维度分别是什么？
一致维度如何复用？
指标口径如何统一？
宽表如何服务分析但不失控？
```

数仓建模的核心不是把表分到不同层，而是让业务过程、事实粒度、维度上下文、指标口径和血缘关系稳定下来。不同业务场景的表名、指标和消费方式会变化，但这些问题始终要回答清楚。
