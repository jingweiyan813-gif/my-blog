> **本文要点**
>
> 数仓任务成功只说明表被产出，不说明数据已经可信。数据质量要回答的是：数据有没有少、有没有错、是不是按时产出、口径是否一致、主键是否唯一、指标波动是否合理。排障要做的也不是简单重跑任务，而是沿着调度、分区、血缘、代码、口径和下游影响逐层定位，并把问题沉淀成长期监控、告警和治理规则。

在数仓架构里，“表跑出来了”和“数据能被业务使用”之间有一段距离。

任务状态是绿色的，只代表计算任务没有报错；分区存在，只代表结果写进了存储；报表能打开，也不代表指标口径正确。真正的数据可用，需要经过质量校验、问题定位、修复回刷和监控沉淀。

前几篇已经讨论了数仓为什么存在、数据如何接入，以及数据进仓后如何分层和建模。本文收束在最后一个架构问题上：当 ODS、DWD、DIM、DWS 和 ADS 都已经建立后，如何证明这些数据资产是可信、可追溯、可排障、可长期使用的。

本文聚焦四个问题：

```text
1. 数据质量到底在校验什么？
2. 不同数仓分层应该检查哪些问题？
3. 表产出异常时，如何系统排障？
4. 如何把一次问题修复沉淀成长期质量规则？
```

不展开具体调度平台、质量平台、存储引擎和内部工具实现。

图 1：数据质量排障不是把任务跑绿，而是从校验、定位、修复到规则沉淀形成闭环。

## 1. 表跑出来，不等于数据可信

数仓里常见的错误判断是：任务成功，所以数据没问题。

但任务成功只覆盖了最外层的计算成功，并不能证明下面这些问题不存在：

| 问题类型 | 典型现象 | 可能后果 |
|---|---|---|
| 数据缺失 | 当日分区行数明显偏低，部分源表未同步 | 指标低估，报表异常下跌 |
| 数据重复 | 主键重复、明细重复、join 后膨胀 | 次数和金额被放大 |
| 空值异常 | 核心字段为空、维度关联失败 | 分组统计失真，下钻不可用 |
| 口径不一致 | 同一指标在不同表中结果不同 | 业务不信任数据 |
| 分区错误 | 事件日期和分区日期错位 | 趋势错位，补数困难 |
| 延迟产出 | 上游晚到，下游按旧数据出报表 | SLA 失效，业务决策滞后 |
| 枚举漂移 | 源端新增状态值，下游未兼容 | 部分数据被误过滤 |
| 维度变化 | 维表更新方式不一致 | 历史归因错误 |

所以数据质量的核心不是“有没有表”，而是“这张表能否被可信地解释和复用”。

一个更工程化的判断标准是：

```text
任务成功
≠ 分区完整
≠ 字段正确
≠ 主键唯一
≠ 指标口径一致
≠ 下游可放心消费
```

数据开发需要把这些条件显式校验出来。

## 2. 数据质量的六类核心校验

行业里常用完整性、准确性、一致性和及时性描述数据质量。落到数仓开发中，还需要补上唯一性和合理性，因为很多事故并不是“数据没来”，而是“来了两遍”“被 join 放大了”“指标波动不符合业务常识”。

| 质量维度 | 要回答的问题 | 常见校验方式 |
|---|---|---|
| 完整性 | 数据是否缺失 | 分区是否存在、行数波动、核心字段非空率 |
| 准确性 | 字段值是否符合事实和规则 | 金额非负、时间合法、状态枚举合法、源端对账 |
| 一致性 | 同一含义在不同表中是否一致 | 指标对账、维度值对齐、上下游汇总一致 |
| 及时性 | 数据是否按 SLA 产出 | 任务完成时间、分区更新时间、上游延迟监控 |
| 唯一性 | 业务主键是否重复 | 主键重复率、去重前后行数差异 |
| 合理性 | 指标波动是否符合业务常识 | 环比 / 同比波动、分布异常、TopN 异常 |

这些维度不是理论分类，而是排障入口。

以支付成功率从 92% 下降到 65% 为例，排查时不能只看成功率这个派生指标，而要拆开看：

```text
支付请求数是否暴涨？
支付成功数是否下降？
失败状态是否新增枚举？
支付渠道维度是否关联失败？
当日分区是否混入了历史补数？
上游是否有延迟或重复写入？
```

指标异常通常是结果，质量校验要回到明细、分区、维度和口径。

## 3. 不同分层的质量重点不同

数据质量不能只在 ADS 或报表层做。越晚发现问题，影响面越大，修复成本越高。

但每一层的职责不同，质量规则也不应该完全一样。

| 层级 | 质量重点 | 不适合做什么 |
|---|---|---|
| ODS | 分区到达、源端行数、字段结构、同步延迟、删除标记 | 不做复杂业务过滤 |
| DWD | 主键唯一、粒度一致、状态标准化、空值规范、业务过程完整 | 不沉淀报表专用口径 |
| DIM | 维度主键、维度值一致、层级关系、历史变化、快照完整 | 不放事实聚合指标 |
| DWS/DWM | 原子指标口径、公共汇总粒度、上下游对账 | 不放临时需求逻辑 |
| DM/ADS | 应用指标一致性、报表结果稳定、消费字段可解释 | 不把公共逻辑散落到多个出口 |

### 3.1 ODS：先保证数据进得来、追得回去

ODS 是入口层，质量重点是接入稳定和源端可追溯。

常见检查：

```sql
-- 检查当日分区是否产出
select count(*) as row_cnt
from ods_trade_order_di
where p_date = '${biz_date}';

-- 检查源端更新时间是否明显滞后
select max(updated_at) as max_source_updated_at
from ods_trade_order_di
where p_date = '${biz_date}';
```

ODS 不建议轻易丢弃异常数据。比如金额为负、状态未知、字段为空，在入口层更好的做法通常是保留并打标。因为这些异常可能本身就是后续排查源端问题的证据。

ODS 层最常见的问题包括：

- 同步任务延迟；
- CDC 漏数或重复；
- 源端新增字段或变更字段类型；
- 分区日期和业务日期混用；
- 删除事件没有正确同步；
- 补数覆盖了原有分区。

### 3.2 DWD：保证明细粒度和业务过程稳定

DWD 是质量问题最应该集中处理的一层。

DWD 不只是清洗字段，更重要的是把业务过程、主粒度、状态口径和异常标记沉淀下来。

常见检查：

```sql
-- 检查明细事实表主键是否重复
select payment_id, count(*) as cnt
from dwd_trade_payment_detail_di
where p_date = '${biz_date}'
group by payment_id
having count(*) > 1;

-- 检查核心字段空值
select
  sum(case when payment_id is null then 1 else 0 end) as null_payment_id_cnt,
  sum(case when user_id is null then 1 else 0 end) as null_user_id_cnt,
  sum(case when pay_amount is null then 1 else 0 end) as null_pay_amount_cnt
from dwd_trade_payment_detail_di
where p_date = '${biz_date}';
```

DWD 的关键是粒度。支付明细表如果定义为“一行一次支付请求”，那同一个支付请求在同一分区内就不应该重复。除非业务明确允许多次状态更新，否则重复记录会直接放大后续汇总指标。

### 3.3 DIM：保证维度上下文一致

维度质量问题经常不会让任务失败，但会让分析结果变差。

例如渠道维表关联失败，报表仍然能出数，只是大量数据落到“未知渠道”；商品类目历史变化处理错误，历史销售额会被归到当前类目；用户等级用当前快照覆盖历史状态，会让历史分层分析失真。

常见检查：

```sql
-- 检查事实表关联维表后的未知维度比例
select
  count(*) as total_cnt,
  sum(case when d.channel_id is null then 1 else 0 end) as unmatched_cnt,
  case
    when count(*) = 0 then null
    else 1.0 * sum(case when d.channel_id is null then 1 else 0 end) / count(*)
  end as unmatched_rate
from dwd_trade_payment_detail_di f
left join dim_payment_channel_df d
  on f.channel_id = d.channel_id
 and d.p_date = '${biz_date}'
where f.p_date = '${biz_date}';
```

维度层需要重点关注：

- 维度主键是否唯一；
- ID 和名称是否一致；
- 层级关系是否闭合；
- 快照分区是否完整；
- 拉链表生效时间是否重叠；
- 维度变化是否需要还原历史。

### 3.4 DWS/ADS：保证指标口径和消费结果一致

DWS/DWM 和 ADS 的质量重点是指标一致性。

同一个支付成功请求数，如果 DWS 算一次，ADS 又重写一次，BI 再加工一次，就很容易出现三个结果。

更稳妥的方式是：

```text
DWD 保留支付明细
↓
DWS/DWM 沉淀支付请求数、支付成功请求数、支付金额等公共指标
↓
ADS 面向报表裁剪维度和展示字段
```

ADS 层可以做应用裁剪，但不应该重复定义核心口径。否则排障时会很难判断问题来自明细、汇总还是报表二次加工。

## 4. 常见数据质量问题如何定位

数据排障需要先缩小范围，再定位根因。不要一上来就改 SQL，也不要先重跑全链路。

可以按下面的顺序处理：

```text
确认影响
↓
确认任务和分区
↓
沿血缘检查上游
↓
对比历史分区
↓
拆解指标口径
↓
定位代码和数据变化
↓
修复、回刷、通知下游
↓
沉淀监控规则
```

### 4.1 先确认影响范围

排障第一步不是找原因，而是确认影响范围。

需要先问清楚：

| 问题 | 目的 |
|---|---|
| 哪张表、哪个分区异常？ | 锁定数据资产 |
| 哪些字段或指标异常？ | 锁定质量维度 |
| 哪些下游报表、接口、任务受影响？ | 决定通知和修复优先级 |
| 异常从哪一天开始？ | 判断是一次性事故还是持续问题 |
| 是否影响对外口径或核心经营指标？ | 决定是否需要升级处理 |

如果不先确认影响范围，很容易把一个局部异常扩大成全链路重跑，或者只修了表却没有通知已经消费错误数据的下游。

### 4.2 再检查任务、分区和上游到达

常见问题要从最基础的产出状态查起：

```sql
-- 查看最近 7 天分区行数波动
select
  p_date,
  count(*) as row_cnt
from dwd_trade_payment_detail_di
where p_date between '${biz_date_minus_6}' and '${biz_date}'
group by p_date
order by p_date;
```

如果当日行数异常下降，先看上游 ODS 是否完整，再看 DWD 是否过滤过多。

如果当日行数异常上升，重点看：

- 是否重复同步；
- 是否补数写入当天分区；
- 是否 join 维表导致膨胀；
- 是否去重逻辑失效；
- 是否源端业务量真实增长。

### 4.3 用血缘定位问题是在上游还是本层

血缘不是治理平台里的装饰信息，它是排障工具。

定位时可以按层级追问：

```text
ADS 指标异常
↓
DWS 公共汇总是否异常？
↓
DWD 明细是否异常？
↓
DIM 维度关联是否异常？
↓
ODS 源端接入是否异常？
↓
源系统是否真实发生变化？
```

如果 ADS 异常但 DWS 正常，问题多半在 ADS 裁剪、二次加工或报表配置。

如果 DWS 异常但 DWD 正常，问题多半在聚合粒度、过滤条件、指标口径或 join 逻辑。

如果 DWD 已经异常，需要继续看 ODS 接入、源端变更、清洗规则和去重规则。

### 4.4 把指标拆成分子、分母和过滤条件

很多指标异常，根因不在指标本身，而在分子、分母或过滤条件。

以支付成功率为例：

```text
支付成功率 = 支付成功请求数 / 支付请求数
```

排查时应该先拆成：

```sql
select
  p_date,
  count(*) as pay_request_cnt,
  sum(case when pay_status = 'success' then 1 else 0 end) as pay_success_cnt,
  case
    when count(*) = 0 then null
    else 1.0 * sum(case when pay_status = 'success' then 1 else 0 end) / count(*)
  end as pay_success_rate
from dwd_trade_payment_detail_di
where p_date between '${biz_date_minus_6}' and '${biz_date}'
group by p_date
order by p_date;
```

如果分母暴涨，可能是重复请求、重试流量、补数或粒度变化。

如果分子下降，可能是状态枚举变化、成功状态未兼容、支付渠道异常或清洗规则误过滤。

如果分子分母都正常，但 ADS 指标异常，问题可能在下游维度裁剪、过滤条件或报表层二次计算。

## 5. 通用排障案例：支付成功率异常下降

问题现象：ADS 报表中支付成功率从 92% 降到 65%，任务没有失败，分区也存在。

这类问题可以按下面的路径排查。

### 5.1 确认异常是否真实存在

先不要直接信报表结果，而是回到公共汇总或明细层验证。

```sql
select
  p_date,
  sum(pay_success_cnt) as pay_success_cnt,
  sum(pay_request_cnt) as pay_request_cnt,
  case
    when sum(pay_request_cnt) = 0 then null
    else 1.0 * sum(pay_success_cnt) / sum(pay_request_cnt)
  end as pay_success_rate
from dws_trade_payment_1d_di
where p_date between '${biz_date_minus_6}' and '${biz_date}'
group by p_date
order by p_date;
```

如果 DWS 结果正常，ADS 报表异常，问题在应用出口层。

如果 DWS 也异常，继续下钻 DWD。

### 5.2 检查明细粒度是否重复

```sql
select
  count(*) as row_cnt,
  count(distinct payment_id) as distinct_payment_cnt,
  count(*) - count(distinct payment_id) as duplicate_cnt
from dwd_trade_payment_detail_di
where p_date = '${biz_date}';
```

如果重复数明显增加，重点看 ODS 是否重复同步、DWD 是否缺少去重逻辑、是否把同一支付请求的多次状态更新都当成了独立请求。

### 5.3 检查状态枚举是否漂移

```sql
select
  pay_status,
  count(*) as cnt
from dwd_trade_payment_detail_di
where p_date = '${biz_date}'
group by pay_status
order by cnt desc;
```

如果源端新增了 `paid_success`、`success_final` 之类的新状态，但 DWD 仍然只把 `success` 识别为成功，就会导致成功数被低估。

这类问题的修复不能只在 ADS 临时兼容。更合理的做法是更新 DWD 的状态标准化规则，并回刷受影响分区。

### 5.4 检查维度关联是否影响过滤

有些报表只统计有效渠道、指定地区或指定产品线。如果维度关联失败，数据可能被下游过滤掉。

```sql
select
  case when d.channel_id is null then 'unknown' else 'matched' end as channel_match_type,
  count(*) as request_cnt,
  sum(case when f.pay_status = 'success' then 1 else 0 end) as success_cnt
from dwd_trade_payment_detail_di f
left join dim_payment_channel_df d
  on f.channel_id = d.channel_id
 and d.p_date = '${biz_date}'
where f.p_date = '${biz_date}'
group by case when d.channel_id is null then 'unknown' else 'matched' end;
```

如果 unknown 比例异常上升，需要检查维表是否延迟、渠道 ID 是否变更、维表快照是否缺失。

### 5.5 修复不是终点

定位根因后，修复动作通常包括：

| 根因 | 修复动作 | 后续沉淀 |
|---|---|---|
| ODS 漏数 | 补同步并重跑下游 | 上游到达和行数波动监控 |
| DWD 去重失效 | 修复去重逻辑，回刷分区 | 主键唯一性监控 |
| 状态枚举新增 | 更新标准化规则，回刷受影响分区 | 枚举合法值监控 |
| 维表延迟 | 补产维表，重跑关联任务 | 维表分区到达监控 |
| ADS 口径重复计算 | 移除二次口径，引用公共指标 | 指标口径治理 |

修复后还要确认下游是否已经消费错误结果。如果报表、接口或数据产品已经对外展示，需要通知下游重新取数或刷新缓存。

## 6. 质量监控应该如何沉淀

一次排障只解决一个问题，质量规则才能降低下一次事故概率。

成熟的数据质量体系通常至少包括三类规则。

### 6.1 基础规则

基础规则覆盖所有核心表。

| 规则 | 示例 |
|---|---|
| 分区到达 | 当日分区必须在 SLA 前产出 |
| 行数波动 | 当日行数相对 7 日均值波动不超过阈值 |
| 主键唯一 | 明细事实表业务主键重复数为 0 |
| 核心字段非空 | 主键、时间、金额、状态等字段非空 |
| 枚举合法 | 状态值必须属于已知集合 |

### 6.2 业务规则

业务规则和具体业务过程相关。

例如交易域：

| 规则 | 示例 |
|---|---|
| 金额合理 | 支付金额不能为负 |
| 状态流转合理 | 已退款订单必须存在支付成功记录 |
| 指标对账 | DWD 汇总金额与 DWS 支付金额差异在阈值内 |
| 维度关联 | 渠道、商品、地区等核心维度关联失败率不能超过阈值 |

### 6.3 波动规则

波动规则用于发现“没有明显错误但不符合业务常识”的问题。

常见做法：

```text
与昨日比
与过去 7 天均值比
与上周同日比
按核心维度拆分看 TopN 变化
对分子分母分别设波动阈值
```

波动阈值不能机械设置。高波动业务、活动期、节假日、版本发布期都可能导致真实业务波动。如果阈值太紧，会产生大量无效告警；如果阈值太松，又发现不了真实问题。

更实用的做法是按表等级和指标等级分层：

| 资产等级 | 监控策略 |
|---|---|
| 核心公共表 | 强监控，失败阻断下游或强告警 |
| 核心指标表 | 强监控，要求口径对账和波动检查 |
| 普通分析表 | 基础监控，异常通知负责人 |
| 临时表 | 不进入长期监控，但要有生命周期 |

## 7. 排障后的治理边界

数据质量治理不是把所有异常数据都清掉。

有些异常应该修复，有些异常应该保留，有些异常应该标记。

| 场景 | 推荐处理 |
|---|---|
| 源端同步漏数 | 补数、重跑、通知下游 |
| 源端真实脏数据 | ODS 保留，DWD 标记或过滤 |
| 新增业务枚举 | 更新标准化规则，补充监控 |
| 下游临时口径 | 不下沉到公共层，保留在 ADS 或临时分析 |
| 历史口径变更 | 明确回刷范围、版本说明和影响下游 |
| 无下游使用的低价值表 | 进入生命周期治理，不长期保留 |

这里已经进入数据资产生命周期管理的范畴：哪些表需要强监控，哪些表只需要基础监控，哪些临时表应该设置生命周期，哪些低价值表可以结合血缘和下游依赖进入归档或下线流程。对于数仓架构学习来说，这部分不需要单独展开成完整数据治理专题，但必须理解它和数据质量、血缘排障是连在一起的。

排障还需要注意两个边界。

第一，不要用重跑掩盖根因。重跑能解决偶发资源失败，但解决不了口径错误、主键重复、维度缺失和源端变更。

第二，不要在下游临时修公共问题。如果 DWD 的状态标准化错了，就应该在 DWD 修；如果 DWS 的公共指标口径错了，就应该在 DWS 修。只在 ADS 或报表层打补丁，会让同一问题在不同出口反复出现。

## 8. 一套可复用的排障清单

最后可以把数据质量排障压缩成一张检查清单。

| 步骤 | 检查项 |
|---|---|
| 1. 定位资产 | 表名、分区、字段、指标、业务方、负责人 |
| 2. 判断影响 | 影响哪些下游表、报表、接口和数据产品 |
| 3. 看任务 | 调度是否成功、上游是否到达、SLA 是否延迟 |
| 4. 看分区 | 分区是否存在、行数是否波动、更新时间是否异常 |
| 5. 看主键 | 是否重复、是否缺失、粒度是否被破坏 |
| 6. 看字段 | 空值、枚举、范围、格式、单位、时区是否异常 |
| 7. 看口径 | 分子、分母、过滤条件、维度粒度是否一致 |
| 8. 看血缘 | 问题在 ODS、DWD、DIM、DWS 还是 ADS |
| 9. 做修复 | 重跑、补数、回刷、代码修复、口径修正 |
| 10. 做沉淀 | 监控规则、告警阈值、变更评审、下游通知 |

这张清单的价值不在于每次机械执行十步，而是在高压排障时避免漏掉关键判断。

## 9. 小结：数仓架构的可信闭环

数据质量是数仓开发从“能出数”走向“可信可用”的关键能力。

任务成功只是起点。真正可用的数据资产，需要在接入、明细、维度、汇总和应用出口各层建立质量规则；出现异常时，要能沿着分区、血缘、口径和代码定位根因；修复之后，还要把问题沉淀成监控、告警、规则库和变更治理。

好的数据开发不是只会写 SQL 建表，而是能回答三个问题：

```text
这张表为什么可信？
出了问题怎么定位？
同类问题下次如何提前发现？
```

能把这三个问题讲清楚，数据才真正从“跑出来的表”变成“可长期使用的数据资产”。

到这里，数据开发系列的第一阶段可以形成一个完整闭环：

```text
为什么需要数仓
↓
数据如何进入数仓
↓
数据进仓后如何分层建模
↓
表产出后如何保证质量、排障和可信消费
```

这个闭环对应的是数仓架构学习中的核心问题：为什么建、怎么接、怎么组织、怎么证明可信。后续的存储治理、指标平台和数据服务都可以作为更细专题继续展开，但不影响这一阶段的数仓架构主线完整性。

## 10. 参考资料 / 延伸阅读

以下资料均为公开可访问资源，适合作为数据质量、数据校验、血缘和可观测性方向的延伸阅读：

- [Apache Griffin](https://griffin.apache.org/)：开源数据质量解决方案，可用于了解批式和流式场景下的数据质量度量、校验和监控。
- [AWS Deequ](https://github.com/awslabs/deequ)：基于 Spark 的数据质量校验库，适合学习完整性、唯一性、分布约束和异常检测等规则如何工程化。
- [Great Expectations](https://docs.greatexpectations.io/)：数据质量规则与校验框架，适合学习 expectation、checkpoint 和数据质量文档化。
- [OpenLineage](https://openlineage.io/)：开放数据血缘标准，适合理解任务、数据集、血缘事件和下游影响分析。
- [dbt Data Tests](https://docs.getdbt.com/docs/build/data-tests)：适合学习唯一性、非空、关系一致性和自定义 SQL 测试如何融入数据开发流程。
