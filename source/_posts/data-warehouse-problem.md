图 1：数仓把业务系统、日志、Trace、对象存储、反馈与标注数据接入分析侧，并组织成面向报表、排查、评测和训练的数据出口。

> **本文要点**
>
> 数据仓库解决的不是“有没有数据”，而是“数据能不能被长期、稳定、可信地用于分析”。它通过多源接入、主题建模、历史沉淀、质量治理和面向场景的数据出口，把业务系统里的原始记录变成可复用的数据资产。对于 AI 应用，这个问题会进一步扩展到 Trace、Feedback、Artifact、评测样本和训练回流数据的组织。

刚接触数据仓库时，很容易把它理解成一个“大一点的数据库”：业务系统里有数据，数仓里也有数据；数据库能查，数仓也能查；最后好像都是写 SQL。

但数据仓库真正解决的，并不是“有没有地方存数据”，也不是“能不能写 SQL 查到数据”。它解决的是另一个更基础的问题：

> 分散在多个业务系统里的数据，能不能被长期、稳定、可信地用于分析？

业务数据库让业务流程跑起来；数据仓库让业务过程被看见、被理解、被复盘、被优化。所以，本文先不展开 ODS、DWD、DWS、ADS 的分层细节，而是先回答一个更基础的问题：

> 为什么业务系统里已经有数据了，还需要数据仓库？

本文主要从离线数仓视角讨论。实际工程中，根据时效要求，也可能建设实时数仓、湖仓一体或近实时数据服务；现代云厂商对[数据仓库](https://cloud.google.com/learn/what-is-a-data-warehouse)的定义，也普遍强调多源数据整合、历史数据分析和面向 BI / 分析应用的能力。

不管形态如何变化，数仓要解决的核心问题始终相似：把分散、异构、变化中的业务数据组织成可信、可复用、可分析的数据资产。

为了让这个问题不只停留在概念层，后面会用一个行业通用的 AI 应用数据链路做例子：当线上已经有用户请求、模型调用、Agent 执行轨迹、工具调用、生成产物和用户反馈时，为什么仍然需要把这些数据建设成数仓能力。

## 1. 业务库里已经有数据，为什么还需要数仓？

业务数据库首先服务业务运行。比如一个交易系统里，数据库要处理的是：

- 订单能不能创建成功；
- 支付状态能不能及时更新；
- 库存扣减会不会出错；
- 多个用户同时操作时，事务是否一致。

这些问题属于 [OLTP](https://aws.amazon.com/compare/the-difference-between-olap-and-oltp/)，即 Online Transaction Processing，在线事务处理。它更关注单次业务操作的准确性、低延迟、高并发和事务一致性。

但分析问题通常长得不一样。分析更关心：

- 最近 30 天用户留存有没有变化；
- 不同渠道的转化路径有什么差异；
- 某个功能上线后，失败率有没有下降；
- 某类 AI 问答 Bad Case 集中出现在哪些场景；
- 不同模型版本、Prompt 版本或工具链版本之间，效果是否有稳定提升。

这些问题属于 [OLAP](https://aws.amazon.com/compare/the-difference-between-olap-and-oltp/)，即 Online Analytical Processing，在线分析处理。OLAP 不是以高频单条事务读写为中心，而是更常见地进行大范围数据扫描、多表关联、聚合计算、明细下钻、多维分析和趋势归因。

所以，业务库和数仓不是替代关系，而是分工关系。

| 对比维度 | 业务数据库 | 数据仓库 |
|---|---|---|
| 核心目标 | 支撑业务运行 | 支撑分析、复盘和决策 |
| 典型负载 | 高频单行读写、短事务 | 批量读取、聚合计算、多表关联 |
| 数据视角 | 业务流程和当前交易状态 | 分析主题、历史变化和统一口径 |
| 设计重点 | 事务一致性、低延迟、高并发 | 可追溯、可复用、口径一致、稳定产出 |
| 使用方式 | 业务系统在线访问 | BI 报表、经营分析、数据服务、模型评测、训练样本 |

业务系统并不是没有历史数据。订单流水、支付记录、操作日志本身都可能长期保存。但这些数据首先服务业务流程：订单能否流转，支付能否完成，状态能否正确更新。数仓关注的是另一件事：把分散在业务过程里的数据，按照统一的实体、时间、口径和粒度组织起来，让它们可以被反复分析和复用。

如果没有数仓，分析通常会变成几种低效模式：

- 研发、产品、运营、分析同学各自写 SQL，各自理解字段，同一个指标出现多个版本。
- 复杂字段每次分析都重新解析，今天能跑通，明天字段结构一变又要重来。
- 线上库只保留当前状态或缺少历史快照，策略上线前后的变化无法准确回看。
- 下游报表、实验分析、Bad Case 复盘、模型评测重复建设中间表，数据成本越来越高，口径却越来越乱。

数仓要解决的，就是把这些一次性的、分散的、不可复用的分析工作，沉淀成稳定的数据链路和数据资产。

## 2. 经典定义：四个关键词对应四类工程问题

[Bill Inmon 对数据仓库的经典定义](https://docs.oracle.com/cd/A84870_01/doc/server.816/a76994/concept.htm)通常被概括为：subject-oriented, integrated, time-variant, non-volatile。中文可以理解为：面向主题、集成、随时间变化、非易失。

有些中文资料也会把后两个词解释为“反映历史变化”和“相对稳定”。这两个说法便于理解，但在技术文章里需要进一步说清楚：

- “非易失”不是说数仓数据永远不能改，而是说它不承担业务库那种高频在线更新；
- “随时间变化”不是简单保存旧数据，而是通过时间维度、快照、分区、版本等设计支持趋势分析和历史追溯。

这四个词不是概念口号，而是四类工程约束。

### 2.1 面向主题：从系统实现视角转向业务分析视角

业务系统的表通常围绕功能模块设计。一个订单链路里，可能会有用户表、订单表、商品表、支付表、退款表、物流表、行为日志表。它们分别支撑不同业务模块。

但分析时，我们关心的往往不是这些系统模块，而是一个个分析主题：

- 用户增长；
- 交易转化；
- 商品履约；
- 退款原因；
- 渠道质量；
- AI 任务成功率；
- 模型回答质量；
- Agent 工具调用稳定性。

“面向主题”就是把数据从系统实现视角，重新组织成分析视角。这也是为什么数仓会按照业务域、数据域、业务过程和分析主题组织数据。数仓不是简单搬运业务表，而是在回答：

> 这批数据后续会围绕什么问题被分析？

### 2.2 集成：从多源异构数据转向统一实体和统一口径

真实业务里的数据通常分散在多个系统里。用户信息可能在用户系统，订单信息在交易系统，支付信息在支付系统，行为日志在埋点系统，客服反馈在另一个系统。AI 产品里还可能存在模型调用日志、工具调用记录、对象存储里的大字段内容、人工标注平台里的反馈样本。

如果每次分析都临时跨系统拼数据，成本会很高，口径也容易发散。

数仓里的“集成”解决的是跨源数据统一的问题：

- 多个业务库的数据如何同步到分析侧；
- 日志、Binlog、消息队列、文件、对象存储数据如何统一接入；
- 不同系统里的用户、订单、会话、请求、任务如何对齐；
- 相同业务含义的字段如何统一命名、类型、编码和时间口径；
- 一条业务链路中的状态变更、操作日志、反馈数据如何串成完整链路。

集成不是把数据堆在一起，而是把它们变成可以被共同理解、共同计算和共同消费的数据。

### 2.3 非易失：让分析建立在稳定、可追溯的数据基座上

业务系统的数据会不断变化。订单状态会更新，用户信息会修改，商品属性会调整，模型配置会迭代，Prompt 模板会升级。但分析需要一个相对稳定的数据基座。

比如今天凌晨产出的昨天分区数据，正常情况下不应该被随意改写。这样分析同学、报表系统和下游数据服务才能基于同一份数据得到一致结果。

这里的“非易失”不是说数仓数据永远不能修改，而是说它不服务业务库那种高频在线 update/delete。进入数仓后的数据通常通过批量装载、增量同步、分区覆盖、快照保留、拉链维护等方式沉淀下来。

如果确实需要重跑、补数、回刷，也应该有明确的数据范围、任务版本、血缘记录和质量校验。否则一张表每次查结果都不一样，或者一个指标上午和下午口径不同，它就很难作为可信资产被复用。

### 2.4 时变性：让数据保留历史，而不只保存当前状态

业务系统并不是没有历史记录，但它保存历史的首要目的通常是支撑业务流程和审计追踪。数仓中的“时变性”更强调面向分析的历史组织能力。

分析常常要看：

- 过去 7 天、30 天、180 天发生了什么变化；
- 某个策略上线前后指标有什么差异；
- 某类问题是偶发，还是持续恶化；
- 某个 Agent 能力在多个版本中是否退化；
- 同一类用户在不同生命周期阶段的行为是否发生变化。

这些问题依赖历史快照、时间序列和版本对比。

在工程实践里，时变性通常通过事件时间、分区字段、周期快照、增量表、拉链表、版本字段等方式体现。没有这些设计，系统里可能只剩“当前状态”，很难回答过去发生了什么、什么时候发生变化、变化前后有什么差异。

## 3. 数仓交付的不是一张表，而是一条可信数据链路

如果把数仓看成一个数据加工体系，一条典型链路可能是：

1. 业务系统 / 日志 / 消息 / 文件 / 对象存储；
2. 采集同步 / CDC / 埋点上报 / 文件接入；
3. ODS 原始层；
4. 明细建模层；
5. 汇总层 / 宽表层 / 指标层；
6. 数据服务 / BI 报表 / 分析 / 评测 / 训练样本 / 产品应用。

业务系统、日志、消息队列、文件和对象存储是原料。采集同步负责把原料接入分析侧。ODS 保留尽量贴近源端的原始数据。

明细建模层按照业务过程和分析粒度沉淀事实。汇总层、宽表层和指标层面向具体分析场景降低使用成本。应用层再把加工后的数据提供给报表、分析、接口、模型或产品系统。

在这条链路里，SQL 只是加工逻辑的一种表达。真正的数据工程工作还包括：

- 数据同步：全量、增量、CDC、消息、文件接入分别如何处理；
- 时间边界：事件时间和处理时间不一致时，分区应该如何归属；
- 迟到数据：T+1 任务产出后，晚到日志如何补入，是否需要回刷历史分区；
- 分区漂移：跨天完成、跨时区上报、0 点前后数据是否会落到错误分区；
- 变更捕获：新增、更新、删除是否都能正确同步，软删除和物理删除如何表达；
- 幂等回刷：补数、重跑、失败重试是否会造成重复写入或错误覆盖；
- 实体一致性：不同来源中的用户、订单、会话、请求如何统一主键和关联关系；
- 数据质量：空值、重复、异常波动、枚举漂移、解析失败如何监控；
- 元数据与血缘：一张表从哪些源表来，下游被哪些报表、任务和服务使用；
- 权限与合规：哪些字段可见，哪些内容需要脱敏，哪些数据不能进入训练或评测；
- 成本与生命周期：历史数据保留多久，冷热数据如何分层，低价值表如何治理。

所以数仓开发不是“把 SQL 写出来”就结束了。真正交付的是一条可调度、可监控、可追溯、可复用的数据链路。

从工程落地看，数仓至少包含四个层面：链路、模型、治理和应用。链路保证数据能进来、能算出来；模型保证数据能被理解和复用；治理保证数据长期可信；应用决定这些数据最终服务报表、分析、数据产品、模型评测还是训练回流。

## 4. 以 AI 应用数据为例：线上交互如何变成分析资产

放到 AI 产品里，数仓要解决的问题会更具体。假设一个 AI 产品上线后，线上系统里已经记录了用户对话、模型调用、工具调用、Agent 执行过程、产物生成和用户反馈。

看起来数据已经存在，但“数据存在”不等于“数据可分析”。

研发和产品真正想分析时，往往会遇到几个门槛。

### 4.1 数据分散：一次交互链路散落在多个系统里

AI 交互数据通常不会只存在一张业务表里。一部分数据可能在关系型数据库中，记录会话、任务、状态、用户、租户等结构化信息。一部分数据可能在 NoSQL 或对象存储里，保存较大的输入、输出、中间产物和附件内容。一部分数据来自日志、消息队列、Trace 系统或反馈链路。还有一部分样本可能来自人工标注平台、评测平台或运营工具。

如果没有统一的数据链路，下游分析只能在多个系统之间临时拼接，既慢，也不稳定。

### 4.2 数据难读：大字段、编码、压缩和对象引用需要解析

AI 产品里的输入、输出和中间产物不一定都是普通 JSON。有些字段可能经过二进制封装、压缩、加密、Base64 编码或分片存储；有些内容可能只是对象存储地址，需要二次读取；有些字段在不同版本里结构不同，历史数据和新数据还要同时兼容。

如果这些解析逻辑只写在一次性脚本里，下游每个分析需求都要重新处理一遍。数仓要做的是把解码、解压、拼接、结构化解析和异常兜底沉淀成稳定任务，让复杂字段变成可读、可查、可关联的数据。

### 4.3 粒度复杂：必须先定义一行数据代表什么

AI 交互链路通常至少包含几层粒度：

下面这些术语不是某个系统的固定表名，而是 AI 应用数据里常见的建模粒度。不同团队的命名可能不同，但建模时都要回答同一个问题：一行数据到底代表什么。

| 粒度 | 含义 | 适合回答的问题 |
|---|---|---|
| Conversation | 一次连续会话或一个工作区入口 | 这次会话整体体验如何，用户是否持续追问 |
| Turn | 用户一次输入及其对应的系统回复 | 用户问了什么，系统回答是否解决问题 |
| Request / Run | 后端一次完整处理链路 | 本次请求整体耗时、成本、状态、最终结果如何 |
| Step / Span | Agent 执行过程中的一个中间步骤 | 哪一步规划、检索、工具调用或代码执行失败 |
| Call | 一次具体的 LLM call、tool call、retrieval call 或 function call | 单次调用的输入、输出、耗时、错误、Token 和成本如何 |
| Artifact | 最终或中间生成的文档、代码、图片、表格、配置或结构化结果 | 产物是否可用，是否被用户采纳或编辑 |
| Feedback | 点赞、点踩、采纳、编辑、投诉、人工标注等反馈 | 用户是否满意，样本是否可进入评测或训练 |

数仓建模时，必须先明确每张事实表的[粒度（grain）](https://www.kimballgroup.com/data-warehouse-business-intelligence-resources/kimball-techniques/dimensional-modeling-techniques/grain/)：一行代表一次会话、一次请求、一次 Agent Step，还是一次具体调用。

粒度混乱会直接导致重复计数、链路断裂和指标口径不一致。比如把 Call 级明细和 Conversation 级指标混在同一张表里，统计会话数时就很容易被调用次数放大。

### 4.4 消费不同：研发、产品、算法需要的表不是同一张

不同使用者关心的粒度不同：

- 研发排查链路时，更关心一次 Request 下面调用了哪些工具、哪一步失败、错误信息是什么、重试了几次、耗时卡在哪里。
- 产品分析体验时，更关心用户原始问题、模型回答、交互路径、最终产物、用户是否采纳、是否继续追问。
- 算法评测和训练数据建设时，更关心 Bad Case、失败类型、人工标注、样本去重、数据集版本、评测集污染和可回流样本。

这时，数仓要做的就不是“把线上表复制一份”，而是面向不同消费场景组织数据出口。

如果把这件事抽象成一条更通用的数仓链路，它不是从“建表”开始，而是按问题一步步推进：

1. **使用边界定义**：明确哪些数据可以采集、哪些字段需要脱敏、哪些样本可以进入评测或训练。
2. **多源数据接入**：把业务库、日志、Trace、对象存储、反馈系统和标注系统里的数据接到分析侧。
3. **原始字段解析**：把嵌套 JSON、大字段、对象引用、编码压缩内容和分片内容处理成可读、可关联、可查询的字段。
4. **多粒度明细建模**：把 Conversation、Turn、Request、Step、Call、Artifact、Feedback 等不同粒度拆清楚。
5. **面向场景的数据出口**：工程排查需要请求级或 Trace 级链路表，产品分析需要会话级行为、反馈与产物表，评测和训练需要样本表、标注表、Bad Case 表和数据集版本。
6. **评测 / 训练 / 分析闭环**：让这些数据出口继续服务评测、训练、产品分析和工程排查。

这条链路的递进关系是清楚的。

落到表设计上，可以先用一个最小映射来理解这条链路。这里的表名只是示例，重点不是命名，而是每一层承担的职责不同：

| 数仓层级 | AI 数据实体 | 示例表 | 一行粒度 | 典型用途 |
|---|---|---|---|---|
| ODS | 原始会话、请求、日志、Trace、对象引用、反馈记录 | `ods_ai_event_log_di` | 一条源端事件或日志 | 保留原始输入，支持追溯和重放 |
| DWD | Request、Step、Call、Artifact、Feedback 明细 | `dwd_ai_call_di` | 一次模型调用、工具调用或检索调用 | 排查失败、统计耗时、成本和错误 |
| DWD | Conversation、Turn、Request 明细 | `dwd_ai_request_di` | 一次请求或一次对话轮次 | 还原用户问题、系统回复和执行结果 |
| DWS | 会话级、请求级、Trace 级宽表 | `dws_ai_request_trace_wide_di` | 一次完整请求链路 | 工程排查、体验分析、链路归因 |
| ADS | 评测样本、Bad Case、训练回流候选集 | `ads_ai_eval_sample_df` | 一个可评测或可回流样本 | 构建评测集、训练集和复盘看板 |

真正难的地方通常不是写一条查询，而是把复杂数据稳定地变成可消费资产。

## 5. 用 AI 案例回看数仓的四个关键词

回到前面提到的数据仓库定义，AI 交互数据链路正好能对应上“面向主题、集成、非易失、时变性”四个关键词。

| 数仓关键词 | 在 AI 交互数据链路里的体现 |
|---|---|
| 面向主题 | 不再按线上系统的存储结构看数据，而是围绕对话复盘、链路排查、工具调用、产物分析、Bad Case 回流等分析主题重新组织数据 |
| 集成 | 把关系型数据库、NoSQL 大字段、对象存储、日志、Trace、反馈链路和标注平台中的数据接到同一条分析链路里，并统一实体、时间、字段和授权口径 |
| 非易失 | 保留原始记录、解析结果、版本信息和回刷边界，使下游可以基于稳定分区、历史快照和可追溯任务复用数据 |
| 时变性 | 通过事件时间、分区、快照、模型版本、Prompt 版本、工具版本和回刷机制保留历史，使后续可以比较不同时间、不同版本、不同场景下的 AI 能力变化 |

这里还有一个容易被忽略的点：AI 产品里的数据质量问题，不一定只是“空值”或“重复”这么简单。它可能表现为：

- 大字段从一种存储迁移到另一种存储后，历史数据和新数据要同时兼容；
- 某个任务跨天完成，明细记录和大字段内容落在不同日期，需要处理回刷和分区边界；
- 字段存在多种编码形态，单一解析逻辑无法覆盖全部样本；
- Trace 链路里有异步任务、重试任务和失败任务，需要明确它们与主请求的关系；
- 底层明细粒度很细，但下游分析需要更容易理解的请求级或会话级视图；
- 合规筛选必须在明细层贯穿全链路，不能等到最后消费时再过滤；
- 训练集、评测集和线上样本如果没有版本管理，后续很容易出现样本污染和结果不可复现。

这些问题都不是单纯靠一次查询能解决的。它们需要被沉淀成稳定的链路、表结构、质量校验、血缘关系和使用说明。所以，数仓在这里提供的能力不是“查到数据”，而是让复杂数据变得可解释、可追溯、可持续消费。

## 6. 边界与取舍：不是所有数据都需要进数仓

数仓不是万能解法。它适合解决长期、重复、跨团队、需要统一口径的数据消费问题，但并不意味着所有数据都应该被建设成完整数仓链路。几类场景可以先不进入完整数仓：

- 一次性排查：只需要临时确认某个线上问题，不会长期复用。
- 小规模实验：样本量很小，链路还不稳定，先用 notebook 或临时表验证即可。
- 强实时决策：需要毫秒级响应，更适合在线服务、流式计算或特征服务。
- 未明确消费方：暂时不知道谁会用、怎么用、多久用一次，贸然建表容易变成低价值资产。

数仓建设的成本不只是存储和计算，还包括口径维护、调度监控、质量校验、权限管理和文档说明。判断一条数据链路是否值得沉淀，通常要看三个问题：

> 是否会被反复使用？是否需要跨团队共享口径？是否会影响分析、评测、训练或业务决策？

如果答案都是“是”，那它就值得从临时查询走向稳定数仓能力。

## 7. 小结：数仓解决的是数据资产化问题

回到标题：数据仓库到底在解决什么问题？我会把它概括成一句话：

> 数据仓库解决的不是“有没有数据”的问题，而是“数据能不能被长期、稳定、可信地用于分析”的问题。

从工程角度看，数仓至少解决六件事：

1. 把分散在多个系统里的数据接入同一条分析链路。
2. 把系统实现视角的数据，重组为面向业务主题的数据模型。
3. 把不一致的字段、编码、时间、实体关系统一成可复用口径。
4. 把易变的业务状态沉淀为稳定、可追溯的历史数据。
5. 把一次性的 SQL 分析沉淀为可调度、可监控、可回刷、可复用的数据任务。
6. 把数据最终交付给报表、分析、数据产品、模型评测、训练样本和业务决策。

业务数据库负责让业务跑起来，数据仓库负责让业务被理解、被分析、被持续优化。对于 AI 应用来说，这个问题会进一步延伸为：线上请求、执行轨迹、模型输出、工具调用、用户反馈和人工标注，能不能被沉淀成评测、训练、排查和产品分析可以持续使用的数据资产。

后续再继续展开分层、建模、数据质量和 AI 数据闭环。
